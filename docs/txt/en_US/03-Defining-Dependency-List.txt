----------------------------------------------------------------------------------------------------
Defining Dependency (Prerequisite) List
----------------------------------------------------------------------------------------------------

In JxMake, prerequisites are defined as other targets that must exist or of which program codes must
be successfully executed before the dependent target's program code can be executed.

On the other hand, dependencies are defined as files that can cause the dependent target's program
code to be (re-)executed (when any of the dependencies is newer than the target).

It means a dependency is always a prerequisite but a prerequisite might not be a dependency (because
a prerequisite does not have to be a physical file).


In the case of C, C++, and Java source code files, the system will try to recursively auto-detect
all the dependencies (header and source files) and then add them to the list of prerequisites.

If there are some dependencies that were missed in the auto-detection process, they can be added
manually from within the 'target' body to the list of prerequisites using the 'extradep' keyword.

These 'extradep' statements must be located at the beginning of the target body before any other
statements. All non-existent paths added using the 'extradep' statements will be discarded from the
final list of prerequisites.

Please note that. by default, non-existent files/paths will not actually be included by either the
auto-detection process or the 'extradep' statements.


Alternatively, dependency lists can be loaded globally from files using the 'depload' and 'sdepload'
keywords.

In this case, the auto-detection process will be disabled for all targets of which dependencies have
been defined from those file.

The dependency lists in a dependency list file must be specified in the similar format as how the
'GNU Make' dependency lists are defined. This format, in turn, is similar to the output generated by
the command: 'gcc -M' or 'gcc -MM', e.g:

    main.cpp.o: main.cpp all.h sys/sysheader1.h sys/sysheader2.h header1.h header2.h header3.h

Please note that, all relative paths will be immediately resolved into absolute paths using the path
of the dependency list file as a reference.


For better performance, instead of using auto-detection, it is recommended to use the dependency
file generator functions '$gdl_*' and then load the generated files using the 'sdepload' keywords.


It is possible to mix modules and includes by calling both '$gdl_cpp_module' and '$gdl_cpp_include'.
However, due to possible conflicts, the dependency list files created using '$gdl_cpp_module' must
always be loaded before those generated by '$gdl_cpp_include'.
